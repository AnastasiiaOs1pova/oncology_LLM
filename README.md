# Система извлечения данных из онкологических историй болезни в JSON

Репозиторий содержит pipeline для преобразования DOCX-выписок и онкологических историй болезни в структурированный `case.json`.

Проект решает задачу безопасного извлечения клинических данных из неструктурированного медицинского текста без «додумывания» фактов. Основной акцент сделан на детерминированную обработку: итоговый JSON формируется только на основе подтверждённых фрагментов текста, валидируется по схеме и сопровождается QC-проверками.

## О проекте

На вход подаётся DOCX-документ с клинической выпиской, в том числе с таблицами.  
На выходе формируется набор артефактов, пригодных для дальнейшей обработки, проверки и интеграции в downstream-системы.

Pipeline включает:

- извлечение текста из DOCX;
- нормализацию медицинского текста;
- построение `coverage layer`;
- rule-based extraction клинических сущностей;
- сборку итогового `case.json`;
- валидацию по JSON Schema;
- генерацию QC-отчётов;
- LLM-review для поиска пропусков без автоматического изменения финальных данных.

## Мой вклад

В рамках этого проекта я разработала pipeline преобразования онкологических DOCX-выписок в структурированный JSON, включая:

- извлечение текста, включая таблицы;
- нормализацию клинического текста;
- coverage layer для фиксации якорей высокой полноты;
- rule-based extraction по словарям и регулярным выражениям;
- сборку итогового `case.json`;
- JSON Schema validation;
- QC-валидацию и формирование отчётов качества;
- LLM-review для поиска возможных пропусков без автоправки итоговых данных.

## Ключевой принцип

При разборе медицинских документов нельзя «додумывать» факты: ошибка в терапии, диагнозе или датах может быть критичной.

Поэтому в проекте используется консервативный подход:

- итоговый `case.json` заполняется детерминированно;
- всё, что не подтверждено текстом, остаётся `null` или пустым;
- ключевые сущности извлекаются только при наличии достаточного текстового основания;
- результат валидируется схемой;
- качество контролируется через `qc_report.json`;
- LLM используется только как reviewer и не изменяет итоговый JSON автоматически.

## Архитектура pipeline

```text
DOCX
 ├─ extract_text.py        -> extracted.txt
 ├─ normalize_med_text.py  -> нормализованный текст
 ├─ coverage_layer.py      -> coverage.json
 ├─ rules/* + builder.py   -> case.json
 ├─ qc_validate.py         -> qc_report.json
 └─ LLM reviewer           -> llm_missing_therapy.json

 ## Основные артефакты

После обработки кейса формируются следующие файлы:

- `extracted.txt` — извлечённый текст документа, включая таблицы;
- `coverage.json` — сырой слой якорей: даты, числа, препараты, биомаркеры, ключевые упоминания;
- `case.json` — итоговый структурированный JSON по схеме;
- `qc_report.json` — отчёт контроля качества;
- `llm_missing_therapy.json` — отдельный отчёт о возможных пропусках терапии, найденных LLM-review.

## Этапы обработки

### 1. Извлечение текста из DOCX

Из документа извлекаются абзацы и таблицы.  
Табличные данные также преобразуются в текст, поскольку значимые клинические сведения часто содержатся именно в таблицах.

### 2. Нормализация текста

Текст приводится к более стабильному виду для последующей обработки:

- выравниваются пробелы и переносы;
- нормализуются символы, тире, кавычки и служебные обозначения;
- приводятся к единому виду частые медицинские сокращения;
- убирается текстовый шум, мешающий корректной работе правил.

### 3. Coverage layer

Формируется промежуточный слой высокой полноты, в котором фиксируются:

- даты и интервалы;
- числовые значения;
- ключевые слова и клинические маркеры;
- кандидаты на препараты и схемы терапии.

Этот слой нужен для контроля покрытия, QC и последующего аудита извлечения.

### 4. Rule-based extraction

Из нормализованного текста по набору правил, словарей и регулярных выражений извлекаются подтверждённые сущности, включая:

- диагнозы;
- биомаркеры;
- историю терапии;
- лучевую терапию;
- процедуры;
- метастазы;
- дополнительные клинические находки.

### 5. Сборка `case.json`

Итоговый JSON собирается по заранее заданной схеме.  
Если факт не найден надёжно или не подтверждён текстом, соответствующее поле не заполняется.

### 6. Контроль качества

После сборки результат проходит:

- валидацию по JSON Schema;
- QC-проверку с уровнями:
  - `blocker`
  - `major`
  - `warn`
  - `info`

### 7. LLM-review

Отдельный модуль анализирует текст и ищет возможные пропуски в терапии.  
Результат сохраняется в `llm_missing_therapy.json`.

**Важно:** этот этап не изменяет итоговый `case.json` автоматически и используется только как дополнительный слой проверки.

## Результаты обработки

После обработки формируются `case.json`, `qc_report.json` и другие служебные артефакты pipeline.

В проекте LLM не используется для свободной генерации итоговых клинических данных.

Причины:

- в медицинском тексте ошибка в терапии, диагнозе или датах критична;
- свободная генерация повышает риск галлюцинаций;
- для production-like pipeline важна воспроизводимость и проверяемость.

Поэтому итоговый `case.json` собирается детерминированно, а LLM применяется только как вспомогательный слой для поиска возможных пропусков.

## Ограничения

- поддерживается только входной формат DOCX;
- система не даёт медицинских рекомендаций;
- покрытие зависит от правил, словарей и текущего набора поддерживаемых сущностей;
- результат подходит для MVP / прототипа / внутреннего контура проверки, но требует контроля качества на реальных кейсах.

## Возможные сценарии применения

Проект представляет собой MVP/прототип pipeline для контролируемого извлечения клинических данных из онкологических документов с акцентом на доказуемость, валидацию и снижение риска галлюцинаций.

Запуск: `\Desktop\oncology> python -m src.main`

Целевые файлы: `case.json` и `llm_missing_therapy.json`, находятся в `data/outputs/case_{порядковый номер}`

В конце `main.py` пример вызова

```python
if __name__ == "__main__":
    input_path = "data/inbound/patient_0007.docx"
    out = extract_case_json(
        input_path=input_path,
        case_id="case_0007",
        clinical_normalize=True,
        use_llm_missing_therapy=True,
    )
    print(f"OK: saved {out}")
```
